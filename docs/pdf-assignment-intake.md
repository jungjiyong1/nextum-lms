# PDF 과제 문항코드 수집·검증

## 브라우저 수집 순서

1. PDF.js 텍스트층에서 페이지와 좌표 순서대로 7자리 코드를 찾는다.
2. 페이지별로 텍스트 코드가 없는 페이지만 렌더링하고 Tesseract.js를 숫자 전용 모드로 실행한다. 따라서 텍스트 페이지와 스캔 페이지가 섞인 PDF도 누락 없이 합친다.
3. OCR 결과도 없으면 서버 작업을 만들거나 파일을 업로드하지 않는다. 교사가 `페이지: 1234567` 형식으로 순서대로 입력한 뒤 다시 진행한다.
4. 텍스트층/OCR에서 얻은 `page`, `bbox`, PDF 순서를 resolve payload와 match item에 보존한다. 직접 입력한 코드의 `bbox`는 `null`이다.

OCR worker, LSTM WASM, 한·영 fast 모델은 외부 CDN에 의존하지 않고 `npm run postinstall`에서 `public/tesseract`로 복사한다. 생성된 런타임 디렉터리는 Git에 넣지 않는다. 배포 자산은 약 16.3MB이고 브라우저 숫자 OCR은 장치에 맞는 core 하나와 영문 모델 등 약 7MB만 첫 실행 때 받는다. 이후 브라우저 캐시와 Tesseract IndexedDB 언어 캐시를 사용한다.

## 답지 차단

- 파일명에 `정답`, `해설`, `답지`, `answer`, `solution`이 있으면 작업 생성 전에 차단한다.
- PDF 텍스트층에 독립된 정답/해설 제목 또는 정답 키워드와 밀집 답안 행이 함께 있으면 업로드 전에 차단한다.
- 서버도 Storage 원본을 직접 파싱해 같은 문서 특성 판정을 수행하므로 클라이언트 판정을 우회할 수 없다.
- 검색 가능한 글자가 거의 없는 이미지형 PDF는 서버가 첫 3페이지만 한·영 OCR한다. 명시적 답지 제목, 또는 18개 이상의 짧은 `문항번호-답` 쌍이 전체 행의 45% 이상인 경우에 차단한다. StudyQ 코드형 답지는 여러 페이지를 검사할 때 그중 두 페이지 이상에서 페이지당 8개 이상의 `7자리 코드-짧은 답` 행이 검출되고, 전체 행의 38% 이상이며 긴 설명·서술 행은 각각 20% 이하일 때만 차단한다.

OCR 판정과 별개로 숫자 코드 결과와 DB crop을 교사가 검토하고 최종 배정 버튼을 직접 누르는 절차는 유지한다.

## 서버 권위 검증

resolve API는 private Storage 원본을 최대 50MB까지 스트리밍하면서 SHA-256을 계산하고, 같은 바이트를 PDF.js로 파싱해 PDF 서명·구조·페이지 수(최대 200)·답지 특성을 확인한다. 브라우저가 보낸 해시나 페이지 수와 다르면 매칭하지 않는다.

성공한 검사의 해시, 페이지 수, Storage 객체 fingerprint를 job metadata에 기록한다. 업로드 grant는 overwrite를 허용하지 않으며 finalize 시 현재 객체 fingerprint와 DB의 `source_pdf_sha256`, `page_count`, 검사 metadata가 모두 일치해야 한다. resolve 함수의 Vercel 실행 한도는 300초다.
