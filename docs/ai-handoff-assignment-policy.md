# Nextum LMS 문제 배정 정책 논의용 맥락

## 0. 당신에게 요청하는 일

이 문서는 Nextum LMS에서 **학생에게 어떤 문제를, 언제, 얼마나, 어떤 근거로 줄지** 논의하기 위한 맥락이다.

코드 구현안보다 다음에 집중해 달라.

1. 교육적으로 타당한 과제 종류와 문제 구성
2. 공통 과제와 학생별 개인화의 적절한 경계
3. 최초 풀이·교정·재확인을 학습 증거로 해석하는 방법
4. 강사가 실제로 운영 가능한 학생 관리 흐름
5. 과제 과다, 잘못된 자동 판정, 반복 훈련 편향을 막는 장치
6. MVP에서 검증할 정책과 이후로 미룰 정책

`[FACT]`는 현재 사실, `[LOCKED]`는 이미 합의한 제품 제약, `[PROPOSAL]`은 비판·수정 가능한 운영안, `[OPEN]`은 이번에 논의할 질문이다.

---

## 1. 제품과 사용자

- [FACT] Nextum LMS는 학원 강사용 운영 앱이다.
- [FACT] Grade App은 학생이 종이 문제지를 푼 뒤 답을 입력하고 채점·오답 학습을 이어가는 앱이다.
- [FACT] LMS가 과제와 대상 학생을 확정하고, Grade App이 학생별 과제·문제 순서와 풀이 기록을 소비한다.
- [FACT] 현재 학원은 `플립수학 종암`, `팩토플러스 2관학원`이다. 최고 관리자는 여러 학원에 접근한다.
- [FACT] 멀티테넌트 Supabase를 사용하며, 문제은행은 소유권과 별개로 학원별 사용 권한을 부여하는 방향이다.
- [FACT] 기술 스택은 Next.js 16, React 19, TypeScript, Supabase Database/Auth/Storage다.

핵심 목표는 점수표를 늘리는 것이 아니라 다음 질문에 빠르게 답하는 것이다.

> 누구에게, 어떤 유형을, 언제, 어떤 문제로 다시 확인하거나 지원할 것인가?

## 2. 문제은행의 현재 상태

2026-07-20 운영 DB 기준이다.

- [FACT] 수학 문제 9,538개가 있으며 모두 검수됨, 이미지 있음, 정답 있음, 난이도 있음, 분석 유형 태그 승인 상태다.
- [FACT] 현재 문제는 객관식이고 문제 본문은 텍스트가 아니라 이미지다.
- [FACT] 출처별 문제 유형 342개, 통합 분석 유형 342개다.
- [FACT] 난이도 단계별 문항 수:
  - 1단계 3,557개
  - 2단계 4,626개
  - 3단계 1,170개
  - 4단계 185개
- [FACT] 유형당 문항 수는 최소 1, 중앙값 21, 평균 27.89, 최대 107개다.
- [FACT] 3문제 미만인 유형이 13개, 5문제 미만인 유형이 27개다.
- [FACT] DB의 9,538개 이미지 자산 행은 콘텐츠 해시 중복 제거로 Storage 파일 9,536개를 공유한다. 누락 파일은 없다.
- [FACT] 이미지를 다시 배치하면 되므로 OCR이나 문제 본문 재작성은 필요 없다.
- [FACT] 현재 정답은 있으나 유형별 개념 설명·대표 풀이·오류 설명 자산은 충분하지 않다.

### 중요한 데이터 한계

- [FACT] 현재 승인된 `equivalence_key`도 342개로, 사실상 유형 전체 단위에 가깝다.
- [FACT] 따라서 지금 데이터만으로는 숫자만 바꾼 동형 문제와 서로 다른 새 문제를 정교하게 구분할 수 없다.
- [PROPOSAL] `analysis_skill_id`는 학습 유형, 별도 `similarity_group`은 동일·동형 문제 묶음으로 분리한다.
- [PROPOSAL] 동형 정보가 준비되기 전에는 서로 다른 `problem_id`만 보장하고, 알려진 중복만 같은 그룹으로 묶는다.

## 3. 현재 구현된 것과 아직 없는 것

### 현재 구현됨

- [FACT] 교재·학년·단원·유형을 탐색하고 유형 범위에서 실제 `problem_id`를 선택할 수 있다.
- [FACT] 선택된 문제, 대상 반·학생 스냅샷을 한 번에 저장하고 과제를 즉시 게시할 수 있다.
- [FACT] Grade App 제출 결과는 세션·문항별 시도·정답 여부로 축적된다.
- [FACT] 외부 PDF의 문제 코드를 DB 문제와 매칭하여 과제를 만드는 별도 흐름이 있다.
- [FACT] 학습 증거 판정 로직과 학습 분석 화면의 기본 구조가 있다.
- [FACT] 운영 DB에는 게시 과제 8개, 과제 문항 130개, 활성 수신자 9명이 있다.

### 아직 구현되지 않음

- [FACT] 유형 선택 후 학생용 문제지 PDF를 자동 조판하는 기능
- [FACT] 게시 전 초안 저장·실제 문제 검수·PDF 검수 단계
- [FACT] 학생마다 다른 문제 목록과 PDF를 갖는 assignment variant
- [FACT] 최근 풀이 이력·취약 유형을 반영한 자동 문제 추천
- [FACT] 학생별 문제지, 통합 정답지, 학생 PDF ZIP의 비동기 생성
- [FACT] 문제지 생성 결과와 Grade App의 문항 순서를 불변 manifest로 연결하는 기능

즉, 아래의 개인화·이미지 조판 설계는 **합의된 개발 방향이지 현재 완성 기능이 아니다.**

## 4. 이미 합의한 제품 제약

- [LOCKED] 강사가 초안과 실제 문제/PDF를 확인한 뒤 한 번만 최종 게시한다. 자동 발송하지 않는다.
- [LOCKED] 첫 출시에서 공통형과 개인화형을 모두 지원한다.
- [LOCKED] 기본 선택 방식은 `유형별 N문제 + 수동 제외/교체`다.
- [LOCKED] 개인화는 두 진입점을 지원한다.
  1. 교사가 공통 유형·난이도·수량을 정하고 학생별 최근 문제만 다르게 선택
  2. 학습 분석이 학생별 취약 유형·난이도를 제안하고 교사가 검토
- [LOCKED] 기본값은 유형당 3문제, 과제 전체 최대 40문제다. 교사가 수정할 수 있다.
- [LOCKED] 최근 30일 안에 배정·풀이한 동일 문제는 우선 제외한다. 부족하면 최근 문제를 오래된 순으로 다시 허용하고 경고한다.
- [LOCKED] 목표 난이도 중심으로 선택한다. 3문제라면 목표 단계 2개와 인접 단계 1개를 기본으로 한다.
- [LOCKED] 통합 분석 유형 보기가 기본이며, 출처·교재별 유형 보기로 전환할 수 있다.
- [LOCKED] 결과물은 학생용 PDF, 교사용 통합 정답지, 학생 PDF ZIP, Grade App 과제다.
- [LOCKED] StudyQ는 과제 생성 시점의 런타임 의존성이 아니다. 문제는 우리 DB와 Storage에서 선택한다.
- [LOCKED] PDF 생성은 Vercel Queue + 비공개 Python Service(Pillow/PyMuPDF)로 처리하고 Supabase Storage에 직접 저장하는 방향이다.

### 이미지 조판의 기본 방향

- [LOCKED] A4 세로 2열이 기본이며 1열로 전환 가능하다.
- [LOCKED] 원본 비율을 유지하고 이미지를 자르거나 문제 중간에서 페이지 분할하지 않는다.
- [LOCKED] 너무 넓거나 큰 문제는 전체 폭 또는 단독 페이지로 자동 배치한다.
- [LOCKED] 문제 번호만 새로 표시하고 StudyQ 코드·DB UUID 같은 내부 식별자는 학생에게 숨긴다.
- [LOCKED] PDF와 DB manifest에 실제 문제 순서·문제 ID·이미지 해시·답안 스냅샷을 고정한다.

## 5. 현재 학습 증거 모델

### 풀이 구분

- [FACT] `independent_new`: 같은 유형의 새로운 문제를 도움 없이 처음 풂
- [FACT] `independent_same_delayed`: 같은 문제를 나중에 다시 독립적으로 풂
- [FACT] `correction`: 틀린 문제를 교정함
- [FACT] `review`: 복습 풀이
- [FACT] `guided`: 해설·힌트·강사 도움을 받은 풀이

### 상태

- [FACT] `recent_confirmed`: 새로운 문제를 서로 다른 날짜에 독립적으로 성공한 근거가 있음
- [FACT] `verification_needed`: 근거 부족, 한 번의 실패, 확인 시점 경과 등으로 새 확인이 필요
- [FACT] `support_candidate`: 서로 다른 문제·날짜에서 독립적인 어려움이 반복됨
- [FACT] `content_gap`: 문제·유형·난이도 정보가 부족해 판단 불가

### 반드시 유지할 해석 원칙

- [LOCKED] 최초 독립 풀이 기록은 나중에 고쳐도 덮어쓰지 않는다.
- [LOCKED] 같은 문제를 고쳐 맞힌 것과 새로운 문제를 독립적으로 맞힌 것을 구분한다.
- [LOCKED] 미응답은 오답이 아니며 학습 상태 계산에서 제외한다.
- [LOCKED] `모름`은 명시적인 부정 증거지만 단순 미제출은 실력 증거가 아니다.
- [LOCKED] 시간이 지났다고 `최근 확인`을 자동으로 실력 미달로 바꾸지 않고 재확인을 권한다.
- [LOCKED] 학생 문제와 콘텐츠 부족을 분리한다.
- [LOCKED] 학생·반·강사 순위나 단일 준비도 점수를 만들지 않는다.
- [LOCKED] 학생·학부모에게 내부 상태를 자동 공개하지 않는다.
- [LOCKED] 시스템은 후보와 근거를 제안하고 강사가 최종 판단한다.

## 6. 현재 검토 중인 교육 운영안

아래는 확정이 아니라 비판 가능한 초안이다.

### 과제 목적 구분

| 목적 | 문제 구성 초안 | 상태 판정 |
| --- | --- | --- |
| 개념 학습·교정 | 풀이 예시 후 유사 문제 2~4개 | 직접 반영하지 않음 |
| 독립 확인 | 도움 없는 새로운 문제 2개 | 핵심 학습 증거 |
| 유지 복습 | 지난 유형 1~2개씩 혼합 | 장기 유지 확인 |
| 시험 대비 | 여러 유형을 섞은 실전 구성 | 범위별 확인 증거 |

### 문제지 구성 초안

- [PROPOSAL] 완전 개인화보다 `공통 코어 + 개인별 꼬리`를 기본으로 한다.
- [PROPOSAL] 일반 과제 예시:
  - 공통 핵심 6~8문제
  - 개인별 확인·보충 2~4문제
  - 이전 유형 유지 복습 2문제
  - 총 10~14문제, 예상 15~25분
- [PROPOSAL] 40문제는 시험 대비·특별 과제의 상한으로만 사용한다.
- [PROPOSAL] 한 번의 자동 개인화에서는 취약 유형을 최대 2개만 다룬다.
- [PROPOSAL] 현행·복습·시험 계획에서 생긴 추천은 학생에게 여러 장으로 보내지 않고 하나의 문제지로 합친다.

### 학습 순환 초안

1. 학습 당일: 설명·예제·교정 연습
2. 2~3일 뒤: 새로운 문제 2개로 독립 확인
3. 14~21일 뒤: 혼합 문제로 유지 확인
4. 시험 전: 실제 범위와 형식에 가까운 혼합 확인

### 지원 후보 처리 초안

반복 오답 학생에게 같은 유형 문제만 많이 주지 않는다.

1. 강사가 근거 문제 또는 대표 풀이 흔적 확인
2. 설명 또는 소그룹 미니 수업
3. 대표 풀이 예시 1개
4. 도움받는 연습 2개
5. 며칠 뒤 새로운 문제 2개로 독립 확인

문제 이미지만으로 확인·복습은 가능하지만, 실제 보충 지도에는 유형별 설명·예제 자산 또는 강사 개입이 필요하다는 전제다.

## 7. 강사 운영 화면의 목표

- [PROPOSAL] 그래프 중심 대시보드보다 `오늘의 조치함`을 우선한다.
- [PROPOSAL] 다음 항목을 서로 분리한다.
  - 미제출·미완료: 과제 관리
  - 오늘 재확인할 학생·유형
  - 반복 어려움이 관찰된 지원 후보
  - 시험 전 확인 대상
  - 반 전체에서 동시에 오답이 증가한 유형·의심 문항
  - 자료 부족·유형 매핑 오류
- [PROPOSAL] 같은 유형에서 여러 학생이 어려움을 보이면 학생별 과제보다 소그룹 보충 수업 후보로 묶는다.
- [PROPOSAL] 현행·복습·시험 대비에서 같은 학생·유형이 중복되어도 활성 조치는 하나만 유지한다.
- [PROPOSAL] 조치에는 담당 강사, 선택한 행동, 다음 확인일, 후속 결과를 남긴다.

학생에게는 `지원 후보`, `미달` 같은 내부 라벨 대신 다음 행동을 표현한다.

> 지난번보다 나아졌습니다. 금요일에 새 문제 2개로 다시 확인합니다.

## 8. 이번에 답해줬으면 하는 핵심 질문

1. [OPEN] 과제 목적을 위 네 종류로 나누는 것이 충분한가? 합치거나 추가할 목적이 있는가?
2. [OPEN] 수학 학원 운영에서 공통 문제와 개인 문제의 이상적인 비율·총량·시간 예산은 무엇인가?
3. [OPEN] 막 배운 유형의 묶음 연습에서 유형 혼합 연습으로 넘어가는 기준은 무엇이어야 하는가?
4. [OPEN] 학생 상태를 확인하기 위해 문제 수, 날짜 간격, 난이도를 어떻게 잡아야 과잉 판정을 줄일 수 있는가?
5. [OPEN] `support_candidate`에게 문제 추가, 개별 설명, 소그룹 수업 중 무엇을 어떤 근거로 제안해야 하는가?
6. [OPEN] 객관식 정답·확신 여부만으로 부족한 정보를 최소한의 학생 부담으로 어떻게 보완할 것인가?
7. [OPEN] 시험 임박, 반복 실패, 재확인 예정, 일반 복습이 충돌할 때 우선순위를 어떻게 정할 것인가?
8. [OPEN] 유형당 문항 부족과 동형 문제 정보 부족을 고려해 자동 배정을 어디까지 허용할 것인가?
9. [OPEN] 학생에게 도움이 되면서도 낙인·과제 과다·강사의 알림 피로를 막는 운영 규칙은 무엇인가?
10. [OPEN] 파일럿 성공을 무엇으로 측정할 것인가? 점수 외에 강사 시간, 추천 채택률, 재확인 성공, 과제량 등을 어떻게 볼 것인가?

## 9. 답변 시 피해야 할 것

- 일반적인 에듀테크 기능 나열, 근거 없는 AI 개인화 찬양
- 배지·순위·게임화 중심의 제안
- 미응답과 오답, 교정 성공과 독립 성공을 합치는 제안
- 학생을 자동으로 `취약`, `미달`, `마스터`로 확정하는 제안
- 문제를 많이 주는 것을 보충 지도와 동일시하는 제안
- 교사 검토 없이 자동으로 과제를 발송하는 제안
- 현재 없는 풀이 텍스트·오답 원인·정교한 동형 문제 정보를 이미 보유했다고 가정하는 제안
- 운영 데이터 없이 지나치게 정밀한 적응형 알고리즘을 확정하는 제안

---

## 10. 원하는 답변 형식

한국어로, 다음 순서가 좋다.

1. 핵심 권고안 5개 이내
2. 과제 목적별 문제 구성·피드백·학습 증거 정책 표
3. 공통형·개인화형·혼합형의 구체적인 기본값
4. 강사의 일일/주간 학생 관리 흐름
5. 자동 추천 규칙과 사람이 판단해야 할 경계
6. 가장 위험한 실패 모드와 방지책
7. MVP 파일럿 설계와 성공 지표
8. 기존 제안 중 동의하지 않는 부분과 이유

가능하면 하나의 학생 사례를 사용해 `최초 오답 → 교정 → 새 문제 확인 → 유지 확인`까지 보여 달라.
